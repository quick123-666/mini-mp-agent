# 工作方法树 — 人话版 (Work Method Tree in Plain Language)

> 写给不熟悉 ontology / knowledge graph 术语的读者。

## 一句话解释

**它就是一个"任务说明书目录",告诉系统:当你给我一个活儿,我有哪些工具可以选、怎么组合、谁负责。**

## 用厨房比喻理解

想象 mini-mp-agent 是一个**只有 1 个员工的小餐厅**:

```
服务员接到单: "来一份鱼香肉丝"
       │
       ▼
  ┌─ 厨师看单 ─┐
  │  想: 这是 main dish (主菜档口)
  │  不是 dessert 也不是 drink
  └─────────────────────┘
       │
       ▼
  查 "鱼香肉丝" recipe (菜谱)
  需要 3 步:
    1. 切肉丝
    2. 调汁
    3. 大火爆炒
       │
       ▼
  大火爆炒这一步要用什么?
  - 锅
  - 火
  - 600°C
  - 铲子
       │
       ▼
  出菜 ✅
```

**这 4 个层次 = 方法树的 4 层**:

| 层 | 厨房比 | 实际含义 | 例子 |
|---|---|---|---|
| **L0** | 档口类型 | 任务入口,先决定走哪条路线 | 主菜 / 汤 / 甜品 |
| **L1** | 菜谱 | 一个完整做菜流程 | "鱼香肉丝做法" |
| **L2** | 菜谱的步骤 | 菜谱里的某一步 | 切肉丝 / 调味 / 大火爆炒 |
| **L3** | 物理工具 | 真正动手用的家伙 | 锅 / 刀 / 火 |

L0 决定"啥档口",L1 拿出"菜谱",L2 执行"步骤",L3 调出"锅铲"。
**18 项技能分布在 4 个层次上**。

## 为什么需要这个树?

没有树的时候,你给系统一个活儿,它**不知道有啥可用的工具**,只能 LLM 临时编一个方案:可能漏掉、可能编出根本不存在的工具。

有了树之后,**所有可用的"技能"被列出来**,LLM 只能从清单里挑,**不能瞎编**:

```
可用工具清单 (18 个):
  入口类 (5):
    - 直接回答一件事
    - 做一次完整的任务
    - 让 5 个人讨论给我答案
    - 自动完整跑一遍规划
    - 自动+把过程记到笔记里
  流程类 (5):
    - 把活儿拆成小步骤
    - 把步骤排成顺序
    - 实际去执行步骤
    - 检查执行结果打分
    - 失败时反思为什么
  步骤类 (5):
    - 从笔记里找有用的信息
    - 把这次对话存到笔记
    - 让一个 LLM 给输出打分
    - 提取文本里的人名/工具/概念
    - 给笔记做健康检查
  工具类 (3):
    - 安全地写一个文件
    - 多件活同时干
    - 判断该不该停下来
```

## 4 个核心问题 — 我怎么问

这个树提供 4 个动作,正好回答 4 个常见问题:

### 问题 1:「我想做的事,你能做吗?」

```
用法: tree.search("我的活儿", 列出前几个)
```

```
问: "并行执行任务"
答: 排前 3 的是 →
  1. 多件活同时干 (50% 相关)
  2. 实际去执行步骤 (30% 相关)
  3. 把步骤排成顺序 (20% 相关)
```

**怎么打分?** 关键词直接命中最相关(+2 分),名字命中次之,描述命中再次。所以搜 "parallel concurrent" 立刻命中"并行执行"。

### 问题 2:「这个东西具体是干啥?」

```
用法: tree.get("execute_task")
```

返一个完整卡片:
```
名字: Execute Task (实际去执行步骤)
层: L1
做什么: 按计划执行步骤
要输入: 计划 + 一个 LLM 函数
输出: 结果列表
依赖: atomic_write, parallel_execute (这俩先跑)
常见失败: 步骤超时 / 步骤崩溃 / 部分完成
谁负责: Worker
```

8 个字段一次给齐,**不靠 LLM 临时组织**。

### 问题 3:「这个东西里面包含什么?」

```
用法: tree.get_children("m_auto", depth=1)
```

```
问: 全套模式 (m_auto) 展开 1 层
答: 5 个步骤全在这 →
  - 拆任务
  - 列计划
  - 执行
  - 打分
  - 反思

展开 2 层:
  - ...上面 5 个 + 它们的子工具
```

L3 的"原子写文件"展开返回空,**因为工具没有下一层**。

### 问题 4:「从这点到那点怎么走?」

```
用法: tree.find_path("m_sprint", "atomic_write")
```

```
问: 全套+记笔记模式 (sprint) 怎么到原子写文件 (atomic_write)?
答: 3 步路 →
  m_sprint → wiki_persist → atomic_write

意思是: 走"存笔记到 wiki"这条分支,会经过"原子写文件"
```

**用途**: 让用户 / LLM 心里有数"这活儿会经过哪些环节"。

## 健康检查 — 一键验证整棵树没坏

```
用法: tree.validate()
```

返 4 件事:
```
1. valid: True/False (树有没有坏)
2. errors: [] (具体错在哪)
3. stats: {节点 18 个, 边 25 条}
4. by_level: {L0:5, L1:5, L2:5, L3:3}
   by_role: {planner:3, worker:3, reviewer:3, reflector:1, dispatcher:5, shared:3}
```

**会查 3 件事**:
- 每条边引用的节点是不是存在
- 节点级别跨度合理(不能从 L0 一跳到 L4,中间没经过 L1/L2/L3)
- 不能出现环(死循环)

**这块对设计师友好**: 如果你不小心把 `atomic_write` 写成 `atomic-write`(拼写错),校验立刻告诉你。

## 它是怎么长出来的

把**18 个技能描述**写成 **15 个小文件**:

```
methods/
├── _index.json         ← 程序读这个启动
├── tree.yaml           ← 人脑看这个理解结构
├── recipes/            ← 每个技能一份
│   ├── m_qa.yaml
│   ├── m_task.yaml
│   ├── decompose_task.yaml
│   ├── execute_task.yaml
│   ├── atomic_write.yaml
│   └── ... 15 个
└── _meta/
    └── graph.json      ← 边列表
```

**关键**:
**YAML 文件里写的英文、人话、字段都是给人看的**。程序启动时把它们转成内存里的 Python 对象,加载 18 个就够,只读一次。

## 一句话总结

它就是 mini-mp-agent 的**办事清单 + 操作手册**:

- **清单**让你不能凭空发明工具
- **手册**告诉你什么活找谁
- **验证**保证清单前后一致
- **查询**让你能问"这活怎么干"

跟工作方法树概念很配:
> 树 = 有结构 (4 层) + 可搜索 (4 个问法) + 可验证 (validate) + 可执行 (集成到 PWR Loop)
